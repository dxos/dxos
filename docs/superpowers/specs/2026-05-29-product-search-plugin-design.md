# Product Search Plugin â€” Design

Date: 2026-05-29
Status: Approved (design); pending spec review.

## Summary

A new Composer plugin (`@dxos/plugin-product-search`, id `org.dxos.plugin.product-search`) that lets a
user perform **structured product searches across multiple vendor sites** â€” general retail
(`amazon.com`) or vertical providers (`autotrader.com`, `mobile.de`, `rightmove.co.uk`).

Each vendor is described by a **Provider template** that captures (a) the typed search
fields the site supports, (b) how to turn those fields into an HTTP request, and (c) how to
extract result listings from the HTTP response. Templates are **generated by an LLM
skill** that analyzes the site once, and are **user-editable**. Executing a search runs
the template **deterministically** (no per-search LLM cost). A search may target multiple
providers; its input form is the **union** of the selected providers' fields.

Results are stored as **linked objects** (stripped metadata + image refs) and viewed in a
**masonry card grid with master/detail selection**, modeled on `plugin-feed`'s
`MagazineArticle`.

The plugin is modeled structurally on `plugin-feed`:

| plugin-feed  | plugin-product-search |
| ------------ | --------------------- |
| Subscription | Provider              |
| Magazine     | Search                |
| Post         | Result                |

## Goals

- Configure vendors as editable Provider templates, mostly authored by an LLM skill.
- Define a `Search` as a typed configuration of what the user is looking for.
- Run a search locally (user-invoked) **or** on the server (agent-scheduled), via Operations.
- Store results as linked objects with stripped metadata and image refs.
- View results as masonry cards with master/detail selection.

## Non-Goals (MVP)

- Authenticated / anti-bot scraping (cookies, captcha, headless browser rendering). MVP
  assumes a simple HTTP fetch returning HTML or JSON. Noted as a follow-up risk.
- A visual template-authoring UI beyond a JSON/form editor over the stored template.
- Saved-search scheduling/cron UI (the operation is scheduleable; UI is a follow-up).
- Cross-provider result de-duplication beyond a simple URL/title key.

## Architecture

Three ECHO types + Operations + a Skill + React surfaces, composed via `Plugin.define`

- `AppPlugin.add*Module` exactly like `FeedPlugin.tsx`.

### Decisions (from brainstorming)

1. **Execution strategy:** LLM authors a reusable template; search execution runs the
   template deterministically. (LLM-at-authoring, not LLM-at-search.)
2. **Search criteria:** typed fields â€” `string | number | number-range | date-range |
boolean` â€” rendered via `react-ui-form`.
3. **Field schema home:** lives on each Provider as JSONSchema. A multi-provider Search
   form is the **union** of its providers' fields. No separate `Category` type.
4. **Mapping representation:** declarative config (no eval). Search-request schema stored as
   `JsonSchema`; request + result mappings stored as structured descriptors run by a
   deterministic interpreter.

## Types (ECHO Schema)

All types use the established pattern: `// @import-as-namespace`, `Schema.Struct`,
`LabelAnnotation`, `IconAnnotation`, `Type.makeObject(DXN.make(...))`, plus convenience
`instanceOf()` / `make()` exports. `SkillsAnnotation.set([SKILL_KEY])` on `Provider`.

### `Provider` (`types/Provider.ts`)

A site template.

```text
name: string
url: string                         // base site URL
description?: string
kind: 'api' | 'scrape'              // declared strategy
icon?/iconHue?

searchSchema: JsonSchema            // typed fields; JSONSchema -> Effect Schema via toEffectSchema()
                                    // field types map to: string | number | number-range | date-range | boolean

request: {                          // declarative request mapping
  method: 'GET' | 'POST'
  urlTemplate: string               // e.g. https://www.mobile.de/...?ms={make}&model={model}
  query?: Record<string, Binding>   // param name -> binding to a searchSchema field
  body?: Record<string, Binding>
  headers?: Record<string, string>
}

result: {                           // declarative result mapping
  responseType: 'html' | 'json'
  itemLocator: string               // CSS selector (html) or JSONPath (json) selecting each listing
  fields: {                         // per-field extractor relative to an item
    title: Extractor
    url: Extractor
    price?: Extractor
    currency?: Extractor
    image?: Extractor               // image url/ref
    [key: string]: Extractor        // additional stripped metadata
  }
  pagination?: { ... }              // optional; MVP may fetch page 1 only
}

enabled: boolean
```

`Binding` = `{ field: string; transform?: string }` (e.g. map a number-range field to two
params). `Extractor` = `{ selector: string; attr?: string }` for HTML or `{ path: string }`
for JSON. Exact shapes finalized during planning; both are declarative and editable.

### `Search` (`types/Search.ts`)

What the user is looking for.

```text
name: string
providers: Ref<Provider>[]                  // multi-vendor target
criteria: Record<string, unknown>           // values for the union of provider fields, keyed by field name
results: Ref<Result>[]                       // linked results (FormInputAnnotation.set(false))
resultState?: Record<resultId, { ... }>      // optional per-result user state (starred/hidden), feed-style side map
status?: 'idle' | 'running' | 'error'
lastRunAt?: string
```

### `Result` (`types/Result.ts`)

One listing.

```text
title: string
url: string
price?: number
currency?: string
images: string[]                    // image refs/urls
provider: Ref<Provider>
search: Ref<Search>
properties: Record<string, unknown> // stripped key/value metadata (or JSON blob)
fetchedAt: string
```

First-class `title`/`price`/`images`/`url` drive the card; `properties` holds the rest.

## Operations (`types/SearchOperation.ts` + `operations/*.ts`)

Defined with `Operation.make({ meta, input, output, services })`, implemented via
`Operation.withHandler(Effect.fn(...))`, aggregated in `OperationHandlerSet.lazy(...)`.

1. **`AnalyzeProvider`** â€” input `{ url }` (and optional hints). Fetches the site, inspects
   its search interface + a sample result page, and produces `searchSchema` + `request` +
   `result`; creates/updates a `Provider`. LLM-backed (skill tool). Services: `Database`,
   web-fetch capability, AI service.
2. **`RunSearch`** â€” input `{ search: Ref<Search> }`. Loads the Search; for each enabled
   provider, invokes `RunProviderSearch`; links produced Results; updates `status`/`lastRunAt`.
   Deterministic. User-invokable and agent-scheduleable (`Operation.invoke` / `Operation.schedule`).
   Services: `Database`.
3. **`RunProviderSearch`** â€” input `{ search, provider }`. Binds `criteria` to the provider's
   `request` (only fields the provider declares), executes the HTTP fetch, runs the `result`
   mapping over the response, and returns/creates `Result` objects. Deterministic. Services:
   `Database`, web-fetch capability.

HTTP fetch (and scraping) goes through the **DXOS edge proxy** rather than fetching
directly from the client â€” avoids CORS and centralizes egress. Confirm the exact edge-proxy
API / capability during planning.

## Skill (`skills/provider-skill.ts`)

`Search Builder` skill keyed on `Provider.SKILL_KEY`:

- `tools: Skill.toolDefinitions({ operations: [AnalyzeProvider, ...] })`
- `instructions`: "Given a vendor site URL, fetch it, determine the supported search
  parameters, the HTTP request format (formal API if present, else the search URL/query
  used by the site), and how each result listing and its fields are located in the response.
  Emit a Provider template (searchSchema + request + result) via AnalyzeProvider."

Registered via `AppCapabilities.SkillDefinition` in `capabilities/skill-definition.ts`.

## UI (`containers/`, modeled on `MagazineArticle`)

- **`SearchArticle`** â€” master/detail layout:
  - **Left (config):** provider multi-select; a `react-ui-form` whose Effect Schema is built
    at render time from the **union** of selected providers' `searchSchema` (merge JSONSchema
    field defs, then `toEffectSchema`); a **Run** button invoking `RunSearch`.
  - **Range fields:** `number-range` / `date-range` are expressed via a **schema annotation**
    on the field (not a distinct primitive type). `react-ui-form` resolves the annotation to a
    range input component (define a custom field component if one does not already exist). The
    skill emits this annotation in the JSONSchema it generates.
  - **Right (results):** `Masonry.Root` of `ResultCard`s; `useSelected()` drives a detail
    pane (full metadata, images, link out).
- **`ResultCard`** â€” Card surface for `Result`: `Card.Poster` (first image), title (line-clamp),
  price, provider name. Mirrors `MagazineTile`.
- **`ProviderArticle`** â€” form/JSON editor over a Provider template so users can hand-edit the
  schema and mappings; "Re-analyze" action re-runs `AnalyzeProvider`.

### App-graph (`capabilities/app-graph-builder.ts`)

- A **Providers** branch under the space listing Provider templates (like the Feeds branch).
- Provider node action: **Re-analyze** (`AnalyzeProvider`).
- Search node action: **Run** (`RunSearch`).

### Create-object (`capabilities/create-object.ts`)

Creating a Search seeds an empty config; creating a Provider opens the analyze flow (or a
blank editable template). Optionally seed one starter Provider for demo, feed-style.

## Plugin composition (`ProductSearchPlugin.tsx`)

```ts
Plugin.define(meta).pipe(
  AppPlugin.addAppGraphModule({ activate: AppGraphBuilder, activatesOn: ... }),
  AppPlugin.addSkillDefinitionModule({ activate: SkillDefinition }),
  AppPlugin.addCreateObjectModule({ activate: CreateObject }),
  AppPlugin.addOperationHandlerModule({ activate: OperationHandler }),
  AppPlugin.addSchemaModule({ schema: [Provider.Provider, Search.Search, Result.Result] }),
  AppPlugin.addSurfaceModule({ activate: ReactSurface }),
  AppPlugin.addTranslationsModule({ translations }),
  AppPlugin.addPluginAssetModule({ asset: { ... PLUGIN.mdl ... } }),
  Plugin.make,
)
```

`meta.ts`: id `org.dxos.plugin.product-search`, name `Search`, icon (e.g. `ph--magnifying-glass--regular`),
`private: true` package, `tags: ['labs']`. A `PLUGIN.mdl` design doc accompanies the plugin
(per project rules, PLUGIN.mdl replaces a separate design doc for plugins).

## Testing

- Unit-test the deterministic mapping interpreter: `criteria + request -> HTTP request`, and
  `response + result -> Result[]` with fixture HTML/JSON (no network).
- Operation tests for `RunSearch` / `RunProviderSearch` using a stubbed fetch service.
- Skill/`AnalyzeProvider` test via `AssistantTestLayer` with a memoized LLM conversation.
- Storybook stories for `SearchArticle`, `ResultCard`, `ProviderArticle` (with `withTheme()` +
  `parameters: { translations }`).

## Risks / Open Questions (resolve during planning)

1. Exact DXOS **edge-proxy** API/capability for HTTP fetch â€” confirm and reuse.
2. Anti-bot / auth on real sites is out of scope for v1; MVP targets simple fetchable HTML/JSON.
3. Merging JSONSchema field defs across providers when names collide (union strategy:
   key by field name; identical names assumed compatible; flag conflicts).
4. Range fields use a **schema annotation** + a `react-ui-form` field component that resolves
   it; confirm whether a range input component already exists or must be added.
